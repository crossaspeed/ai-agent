压测的优化步骤：
- 项目最开始进行压测的QPS的大致数值为:300-400之间
1. mongodb添加索引
```
# 进入docker里面的mongodb
docker exec -it ContainerId mongodbsh
# 查找相关数据库
show dbs
# 查找Collections
show collections
# 添加索引
db.chat_message.createIndex({ "memoryId": 1 })
# 验证索引是否创建成功
db.chat_message.getIndexes()
```
QPS提升至700

进行操作：

1. 当线程数量从100上升到200的时候，QPS没有明显的变化，但是各个接口的响应时间都上升为原来的两倍
2. 调整MongoDb连接的数量（maxPoolSize）（由原来的120上升为200）和Tomcat的线程数量（由原来的200上升为400）
3. 出现新的问题：
   1. 当线程数量从100上升到200的时候，响应时间不仅没有变化，而且QPS反而下降了
   2. 原因：当线程数量从100上升到200的时候，Java产生对象的速度翻倍。如果JVM内存不够大，JVM就会频繁的出发Full GC
4. 发现问题：
   1. 原来的JVM最大的堆内存只有892MB
5. 解决问题：
   1. 调整最大的堆内存为1.5G
6. QPS稳定在700，并且当线程数量变化的时候，大部分接口的响应时间没有太大变化。

新增操作：
1. 会话文档新增字段：title、content、createdAt、updatedAt
   1. 新增索引：memoryId，updatedAt（降序索引）

   1. 之前的操作：获取消息是getAllSessions()，标题是在线解析的（mongodb数据库里面Session没有设置title字段），没有进行分页操作，全量获取
   2. 改进的操作：查询数据的时候只查询title和memoryId，并且执行分页查询的操作，每次查询只最多查询20条数据
